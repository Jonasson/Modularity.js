Introduction

One big challenge when developing JavaScript heavy web sites/applications is to organize them in a good way. Modularity.js approaches this problem with the help of a couple of great JavaScript projects and then adds some own ideas on top of that.

There are two core ideas Modularity is built upon worth mentioning: modularity and declarative programming.

Modularity

Modularity is such an important idea, it gave this project its name. The hope is that modularity will let you organize your code in a way that both lets you reuse more code, and to avoid that wall we all have run into. The wall that hits you in your face when your project have been a headache to continue with. When all the fun ends.

Declarative Programming

The definition of declarative programming is maybe a bit vague. When we talk about Modularity.js we essentially mean that you tell when a module should be used, and maybe set some options, via simple data-attributes directly in the markup. When you create a module, you define a custom data attribute associated with it. In the example below there are modules associated with the "data-controller", "data-article" and "data-menu" attributes. 

<html lang="en">
  <head>
    <title>Modularity.js Example</title>
    .
    .
    .
    <script data-main="scripts/bootstrap.js" src="scripts/require.js"></script>
  </head>
  <!-- The routes below is handled by a Backbone.js controller -->
  <body id="controller" data-controller='{
    ""            : [ "welcome:article" ],
    "about"       : [ "about:article" ],
    "inspiration" : [ "inspiration:article" ]
  }'>
    <div id="page">
      <!-- this module turns the menu into a dropdown menu.
           It's unaffected by the controller. -->
      <nav data-menu='{ "effect": "slide" }'>
        <ul>
          <li><a href="#">Welcome</a></li>
          <li><a href="#about">About</a></li>
          <li><a href="#inspiration">Inpsiration</a></li>
        </ul>
      </nav>

      <!--
        These divs are bound to an article module. The controller decides if
        they should be visible or not.
        The module checks the div id to decide which content to load.
      -->
      <div id="welcome" data-article='{}'></div>
      <div id="about" data-article='{}'></div>
      <div id="inspiration" data-article='{}'></div>
    </div>
  </body>
</html>

About this documentation

This site is built with Modularity.js, and is in itself an example of how Modularity.js works. Please, take a look at the source if you are curious and want an example of how it works.

Just as the project itself, this documentation is in a rough state. Expect a lot of Swenglish and quirky behavior...

Dependencies

The goal has been to minimize the dependencies to a minimum.

RequireJS

RequireJS allows you to divide your project into smaller parts in a clever way. You could use Modularity.js without RequireJS if you want to, but it is not meant to be used that way.

jQuery

Great DOM manipulation. There is a hard dependency on this framework and Modularity.js will not work without it.

Other inspiration

  • Zakas modular architecture

    This video is probably what got it started, even though Modularity.js is quite different from the code Zaka shows here. The sandbox was a part of Modularity.js in its earliest incarnations, but was eventually dropped. You could probably offer similar functionality via RequireJS.

  • Knockout

    The way Knockout handles attribute values is beautiful. Modularity.js uses JSON strings which is not optimized to be embedded in HTML attributes, but you must start with something...
